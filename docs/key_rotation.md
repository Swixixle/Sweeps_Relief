# Key rotation (receiving side)

Relief verifies signed artifacts from Sweeps_Intel using public keys listed in `trust_store.json` at the repo root. When Intel rotates one of its signing keys, Relief's trust store must be updated to trust the new key. This document covers that receiving-side workflow.

For the sending side (Intel generating a new key and handing the public key off), see `docs/key_rotation.md` in Sweeps_Intel.

Relief's own internal signing keys (policy.json, log bundles) have a separate lifecycle that is not covered by this document. Those use a different envelope format; see TODO.md for planned migration.

## When Intel rotates a key

Intel will produce a new public key (via its own rotation procedure) and hand off a JSON trust-store entry generated by Intel's `scripts/trust_store_entry.py`. The handoff is currently manual — text message, secure chat, or any channel the operator verifies by out-of-band means.

## Procedure

1. **Verify the handoff.** The JSON entry should look like:
   ```json
   {
     "key_id": "intel-snapshot-key-v2",
     "algorithm": "ed25519",
     "public_key_pem": "-----BEGIN PUBLIC KEY-----\n...\n-----END PUBLIC KEY-----\n",
     "issued_at": "2026-05-21T00:00:00Z",
     "authorized_for": ["intel_snapshot"],
     "revoked_at": null,
     "revocation_reason": null
   }
   ```

   Confirm by out-of-band channel (phone, in-person, etc.) that the public key is what Intel actually produced. Do NOT skip this step — the whole point of the trust store is to anchor trust; trusting a key you received via an unverified channel defeats the purpose.

2. **Add the entry to `trust_store.json`.** Append it as a new object in the `keys[]` array. Do NOT remove the existing `-v1` entry yet — both keys must remain trusted during the grace period so in-flight signed artifacts from Intel still verify.

3. **Update `updated_at`** on the trust store to the current UTC timestamp.

4. **Validate the JSON:**
   ```bash
   python -c "import json; json.load(open('trust_store.json'))" && echo "OK"
   ```

5. **Test verification** with a signed artifact from Intel using the new key, if one is available. Otherwise, a synthetic test via pytest is sufficient confirmation that the trust store loads correctly.

6. **Commit the trust store update:**
   ```bash
   git add trust_store.json
   git commit -m "trust: add intel-snapshot-key-v2 during Intel's key rotation"
   git push
   ```

7. **Revoke the old key when Intel says it's safe.** After Intel's grace period expires, Intel will notify you that the old key is being retired. Mark it revoked in your trust store:
   ```json
   {
     "key_id": "intel-snapshot-key-v1",
     "revoked_at": "2026-06-21T00:00:00Z",
     "revocation_reason": "Intel scheduled rotation to v2"
   }
   ```

   Commit this change separately from the key addition. Revocations are audit-visible.

## What NOT to do

- **Do not trust a key received via an unverified channel.** The whole point of the trust store is that it anchors trust; trusting handoffs you haven't verified out-of-band undermines the chain.
- **Do not delete old key entries from `trust_store.json`** — revocation preserves audit history. Set `revoked_at` and `revocation_reason`, never remove entries.
- **Do not revoke early.** If you revoke Intel's old key before Intel has stopped signing with it, you break your ability to verify Intel's output mid-rotation.
- **Do not confuse this with Relief's internal signing rotation.** That's a different workflow using different code paths (TODO.md Option C).
